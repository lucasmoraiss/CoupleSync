# ADR-003: Use In-Process Background Jobs For Pluggy Sync And FCM Dispatch In V1

## Context
CoupleSync requires periodic transaction sync and event-driven alerts. Scale is limited to a small pilot, so complex distributed job infrastructure is out of scope.

## Decision
Run background workloads inside the backend process using hosted services:
- Scheduled Pluggy sync polling for active integrations.
- Post-sync alert evaluation.
- Notification dispatch worker with retry and dead-letter status.
Persist job run metadata in PostgreSQL and design all handlers to be idempotent.

## Alternatives Considered
- External queue + worker cluster
  - Rejected: unnecessary infrastructure complexity at pilot scale.
- Cron-only scripts outside API runtime
  - Rejected: weaker observability and more operational drift.
- Manual sync only (no background automation)
  - Rejected: does not meet product expectations or acceptance criteria.

## Consequences
- Positive
  - Simple deployment and easier debugging.
  - Meets AC requirements for automated sync and notifications.
  - Clear migration path to external workers later.
- Negative
  - Single-instance throughput ceiling.
  - Requires careful overlap control and retry limits.

## Notes
If pilot usage grows, extract background handlers to a separate worker process first, then introduce queue infrastructure.
