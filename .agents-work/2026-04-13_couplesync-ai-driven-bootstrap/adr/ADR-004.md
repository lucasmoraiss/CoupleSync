# ADR-004: Use Android Notification Listener For Bank Transaction Capture In V1

## Context
The previous architecture assumed Open Finance provider integration and periodic backend polling for transactions/balances. Product direction changed: CoupleSync V1 will capture bank transaction data from Android push notifications on the user device, parse transaction details locally, and upload structured events to backend. Pilot constraints (up to 10 users), delivery speed, and cost sensitivity favor a simpler integration path without third-party API dependencies.

## Decision
Adopt Android `NotificationListenerService` as the bank integration mechanism for V1.
- Mobile captures notifications from supported bank apps and parses them locally via bank-specific regex patterns.
- Mobile sends authenticated `TransactionEvent` payloads to `POST /api/v1/integrations/events`.
- Backend `NotificationCapture` module treats all client data as untrusted: validates schema, sanitizes strings, enforces couple scope from JWT claims, deduplicates events, and persists normalized transactions.
- Automatic real-time bank balance sync is not available; balances are manual or estimated from transaction history.
- Retroactive transaction history is not available; capture starts from install/permission grant date.

## Alternatives Considered
- Open Finance provider (Pluggy/Belvo/Plaid)
  - Rejected: external dependency cost/complexity, credential/token lifecycle overhead, and slower integration for pilot timeline.
- SMS parsing integration
  - Rejected: lower consistency, telecom/provider variability, and weaker UX compared to app notifications.
- Manual transaction-only entry
  - Rejected: reduced automation and lower adherence to product value proposition.

## Consequences
- Positive
  - Faster and lower-cost path to pilot delivery.
  - No raw banking credentials collected or stored.
  - Near real-time ingestion when notifications are delivered.
- Negative
  - Coverage depends on user-enabled notification permission and bank notification format stability.
  - No retroactive history and no authoritative real-time account balance.
  - Requires ongoing maintenance of bank-specific regex patterns.

## Notes
This ADR supersedes Open Finance assumptions for V1 architecture and module contracts. Security controls for event ingestion are mandatory: payload validation, sanitization, deduplication, rate limiting/anti-abuse guards, and strict couple-level authorization on every write/read path.
