# ADR-001: Adopt A Modular Monolith With Explicit Domain Modules

## Context
CoupleSync V1 must ship quickly for a pilot of up to 10 users while keeping the codebase maintainable and secure. The selected stack is fixed (Expo React Native, .NET 8 Web API, PostgreSQL, Pluggy, FCM), and V1 explicitly excludes microservices.

## Decision
Use a single deployable .NET 8 backend as a modular monolith, split into bounded modules: Auth, Couple, AccountIntegration, Transaction, Goal, CashFlow, and Notification. Keep strict layer separation (API, Application, Domain, Infrastructure) and module contracts via DTOs/events.

## Alternatives Considered
- Microservices per domain module
  - Rejected: unnecessary operational overhead for pilot scale and slower iteration.
- Single project with no module boundaries
  - Rejected: faster short-term setup but high risk of coupling and unclear ownership.
- Backend-as-a-service only
  - Rejected: reduced control for domain-specific Open Finance sync and security rules.

## Consequences
- Positive
  - Faster delivery with one deployment unit.
  - Lower infrastructure complexity while retaining internal boundaries.
  - Clear module ownership supports planner decomposition and future extraction if growth requires it.
- Negative
  - Internal discipline is required to avoid cross-module coupling.
  - A single runtime fault can impact all modules.

## Notes
This ADR aligns with V1 non-goals (no microservices) and pilot-scale constraints while preserving an upgrade path to split services later if needed.
