# ADR-002: Enforce Couple-Scoped Authorization At Token, Service, And Query Layers

## Context
The highest project risk is privacy and cross-couple data leakage. Requirements mandate couple-level data isolation on every endpoint and query.

## Decision
Enforce couple scoping as a defense-in-depth policy:
- JWT includes `couple_id` claim after pairing.
- Controller policies require authenticated user and validate requested resources belong to claim scope.
- Application services receive user context with `user_id` and `couple_id`.
- Repository queries always filter by `couple_id`.
- Financial tables store `couple_id` explicitly for direct scope filtering and auditing.

## Alternatives Considered
- Controller-only scoping checks
  - Rejected: easy to bypass by mistakes in deeper layers.
- Database row-level security (RLS) only
  - Rejected for V1: strong option but adds migration and operational complexity for pilot speed.
- User-level only ownership checks
  - Rejected: does not satisfy shared couple workspace model.

## Consequences
- Positive
  - Strong containment against accidental cross-couple access.
  - Auditability of scope decisions.
  - Explicit pattern reusable across all modules.
- Negative
  - Additional boilerplate in repository contracts and tests.

## Notes
RLS can be revisited post-pilot as a hardening step once delivery risk is reduced.
