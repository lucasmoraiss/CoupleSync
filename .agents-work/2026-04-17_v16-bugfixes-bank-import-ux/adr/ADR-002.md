# ADR-002: Toast/Snackbar Component ‚Äî Custom Implementation vs Third-Party Library

**Session:** 2026-04-17_v16-bugfixes-bank-import-ux  
**Date:** 2026-04-17  
**Status:** Accepted  
**AC:** AC-607, AC-608

---

## Context

V1.5 surfaces all mutation errors via `Alert.alert()` ‚Äî a native Android system dialog. Issues:
- Visual inconsistency: uses system modal style, not app theme colors.
- No recovery affordance: generic "OK" button with no navigation action.
- `COUPLE_REQUIRED` (403) is indistinguishable from a server crash.
- No auto-dismiss; interrupts the user flow.

V1.6 must replace all `Alert.alert()` calls in mutation `onError` handlers with a styled, auto-dismissing, accessible toast pattern. The toast must:
- Use app theme tokens (`colors.error`, `colors.success`, `colors.info`).
- Auto-dismiss after 4 seconds; tap-dismissable.
- Support `role="alert"` and `accessibilityLiveRegion="polite"` for accessibility.
- Render above the bottom tab bar without obscuring primary actions.
- Expose `toast.error()`, `toast.success()`, `toast.info()` imperatively from any component.

---

## Decision

Implement a **custom `ToastProvider` + `useToast` hook** using React context and `Animated` from React Native.

The provider is placed in `app/_layout.tsx` (the root layout ‚Äî above the tab navigator). This ensures toast messages are visible from any tab or modal, including nested navigators.

**Implementation shape:**
- `ToastProvider.tsx` ‚Äî manages a `toast: { message, variant, visible }` state; exposes `show()` via context; renders `<ToastOverlay />` with `position: absolute` and high `zIndex`.
- `Toast.tsx` ‚Äî visual layer using `Animated.View` for slide-in/fade animation; uses `colors.error` / `colors.success` / `colors.info` from `mobile/src/theme/index.ts`.
- `useToast.ts` ‚Äî thin hook: `const { toast } = useToast()`.
- Queue behaviour: if a new toast is shown while one is active, the current toast is dismissed immediately and replaced; no queue stacking for V1.6.

---

## Alternatives Considered

### A. `react-native-toast-message` (third-party, MIT)
- **Pros**: Battle-tested, good docs, handles queue and animation, Expo-compatible.
- **Cons**: Default visual style does not match app theme tokens without custom renderer; adds a dependency (~10 KB) for a component that can be trivially implemented; requires wrapping in a custom renderer anyway for `COUPLE_REQUIRED` special case.
- **Rejected**: The total customization surface equals a from-scratch implementation. Avoiding an extra dependency given the simple requirements is preferable.

### B. `react-native-snackbar` (community library)
- **Pros**: Native Android `Snackbar` UX.
- **Cons**: iOS not needed per V1 scope, but the package adds native module bridging even for Android-only ‚Äî complicates Expo managed workflow; does not support custom theme tokens.
- **Rejected**: Native module overhead unjustified for a component we can build in < 80 lines of React.

### C. Keep `Alert.alert()` + overlay for COUPLE_REQUIRED only
- **Pros**: Zero new code for most cases.
- **Cons**: Does not satisfy AC-607 or AC-608; inconsistent UX remains.
- **Rejected**: Does not meet acceptance criteria.

---

## Consequences

- **Positive**: Full control over styling, animation, and special-case logic (e.g., `COUPLE_REQUIRED` toast with navigation button).
- **Positive**: No new npm dependency.
- **Negative**: Maintainer must implement accessibility attributes (`accessibilityLiveRegion`, `accessibilityRole="alert"`) manually.
- **Note**: If the design system expands significantly in V2, migrating to a design-system toast library is straightforward ‚Äî the `useToast` hook signature is stable.

---

## Notes

- `ToastProvider` must be placed **above** `<Stack>` / `<Tabs>` in `app/_layout.tsx`, not inside them ‚Äî otherwise toasts inside a modal navigator may be clipped by the modal's own z-stack.
- Bottom padding on the `ToastOverlay` must account for the tab bar height (60 dp) plus safe area insets (`useSafeAreaInsets()`).
- `COUPLE_REQUIRED` special casing: the `useToast` hook or an `ApiErrorToast` wrapper detects `error.code === 'COUPLE_REQUIRED'` and appends a navigation button to the toast body rather than emitting a standard error toast.
