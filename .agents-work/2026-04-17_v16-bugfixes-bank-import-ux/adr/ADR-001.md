# ADR-001: Local PDF Bank Statement Parser — PdfPig + Strategy Pattern

**Session:** 2026-04-17_v16-bugfixes-bank-import-ux  
**Date:** 2026-04-17  
**Status:** Accepted  
**AC:** AC-603, AC-604, AC-605, AC-606

---

## Context

V1.5 used Azure Document Intelligence (`prebuilt-receipt` model) to extract data from uploaded bank statement PDFs. The model is designed for retail receipts, not tabular bank export formats — it returns zero fields for every Brazilian bank statement tested. This is a P0 blocker (AC-603 through AC-606).

Constraints driving the decision:
- Free pilot (≤ 10 users) — no paid OCR services.
- Pure .NET environment — no native binary dependencies in Docker image.
- Supported banks: Inter, Nubank, Itaú, Santander, Banco do Brasil, Mercantil, Caixa.
- Azure Document Intelligence must be retained but disabled (rollback path).

---

## Decision

Replace `AzureDocumentIntelligenceAdapter` as the default `IOcrProvider` with `LocalPdfParserProvider`, a pure .NET class that:

1. **Extracts text** from PDF using **PdfPig** (Apache 2.0, no native dependencies, pure C#).
2. **Detects the bank** via a `BankDetector` that iterates registered `IBankStatementParser` implementations and calls `CanParse(extractedText)` — first match wins.
3. **Parses transactions** by dispatching to the matched bank-specific regex parser.
4. **Returns** a JSON string with discriminator `{ "provider": "local-pdf", ... }` compatible with the existing `IOcrProvider` contract.

`OcrProcessingService.ParseCandidates()` is extended with a discriminator check: if `provider == "local-pdf"`, use `ParseLocalPdfCandidates()`; otherwise, fall through to the existing Azure format parser (unchanged).

A feature flag `USE_LOCAL_PDF_PARSER` (default `true`) controls which implementation is registered for `IOcrProvider` in `DependencyInjection.cs`. `AzureDocumentIntelligenceAdapter` is retained for rollback.

---

## Alternatives Considered

### A. Azure Document Intelligence — `prebuilt-layout` or custom model
- **Pros**: Already integrated; could work with bank tables if the right model is chosen.
- **Cons**: `prebuilt-layout` is paid (no free tier at scale); training a custom model requires labeled data, time, and continued cost. Does not satisfy the free-tier constraint.
- **Rejected**: Cost and model preparation effort exceed V1.6 scope.

### B. Google Cloud Vision OCR (existing fallback)
- **Pros**: Already coded as a fallback adapter; free tier (1,000 units/month).
- **Cons**: Returns raw text without structured field extraction; Brazilian bank statement layouts vary dramatically — applying generic regex to Vision output is equivalent to Option C but with an added API call and network latency. Low confidence threshold for tabular data.
- **Rejected**: Extra network call, no structural advantage over local parsing.

### C. Tesseract.NET for image-based PDFs
- **Pros**: Handles scanned (image-only) PDFs.
- **Cons**: Native `libtesseract.so` binary dependency — requires OS-level install in Docker, increases image size significantly, complicates CI. Brazilian bank exports are almost universally text-based digital PDFs.
- **Decision**: Tesseract is deferred. If text extraction returns < 50 characters, `LocalPdfParserProvider` throws `OcrUnavailableException` with message directing the user to use a digital export. Tesseract integration is a future enhancement.

### D. Change IOcrProvider contract to return structured types instead of JSON string
- **Pros**: Cleaner type safety; avoids dual-format JSON parsing in `OcrProcessingService`.
- **Cons**: Breaking change to existing `IOcrProvider` interface, `AzureDocumentIntelligenceAdapter`, `OcrBackgroundJob`, and all integration tests. Significant refactor for a pilot release.
- **Rejected**: Discriminated JSON is a minimal-impact bridge that preserves backward compatibility. The interface contract can be revisited in V2.

---

## Consequences

- **Positive**: Zero ongoing cost; fully offline processing; no Azure dependency for the default path; straightforward unit testing with text fixtures.
- **Positive**: Adding a new bank requires only one new class implementing `IBankStatementParser` — no other changes.
- **Negative**: Parser accuracy depends on bank format stability. Each bank may change PDF layouts with software updates. Regex parsers will need maintenance.
- **Negative**: Scanned/image-only PDFs are not supported in V1.6. Users with scanned statements receive a clear error.
- **Risk**: PDF format changes break a specific bank parser silently (wrong transactions, not an error). Mitigation: parsers should return 0 transactions if the expected section header is not found, triggering `NO_TRANSACTIONS_FOUND` error (EC-003) rather than silently returning garbage data.

---

## Notes

- PdfPig NuGet package: `UglyToad.PdfPig` — Apache 2.0 license, no native dependencies, supports .NET 8.
- Bank keyword detection strategy: each parser's `CanParse()` checks for bank-specific header text (e.g., `"Banco Inter"`, `"Nu Pagamentos"`, `"Banco Itaú"`, `"Banco do Brasil"`) using `string.Contains()` with `OrdinalIgnoreCase`.
- All bank parsers must be registered in `DependencyInjection.cs`; `BankDetector` receives `IEnumerable<IBankStatementParser>` via constructor injection and evaluates them in registration order.
