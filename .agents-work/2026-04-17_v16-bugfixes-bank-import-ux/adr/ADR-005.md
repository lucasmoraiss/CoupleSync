# ADR-005: Quick Income Endpoint — PATCH /budget/income Behavior When No Plan Exists

**Session:** 2026-04-17_v16-bugfixes-bank-import-ux  
**Date:** 2026-04-17  
**Status:** Accepted  
**AC:** AC-611

---

## Context

V1.6 adds a `PATCH /api/v1/budget/income` endpoint so the Dashboard's Quick Income modal can update the gross income for the current month without navigating to the full budget screen. A small but important question: what should happen if no `BudgetPlan` exists for the current month when the user taps "Atualizar renda"?

Options: return 404, require the user to create a plan first; or auto-create a plan with zero allocations.

---

## Decision

**Auto-create a monthly plan if none exists.** The `PATCH /budget/income` endpoint delegates to `BudgetService.UpdateIncomeAsync(coupleId, grossIncome, currency, ct)`, which:
1. Calls `GetByMonthAsync(coupleId, currentMonth)`.
2. If `null` → calls `UpsertPlanAsync(coupleId, currentMonth, grossIncome, currency)` to create a new plan with zero allocations.
3. If found → calls `plan.Update(grossIncome, currency)` and saves.

The endpoint always returns `200 OK` with `{ planId, month, grossIncome, currency }` — it never returns 404.

**Currency default**: if `currency` is omitted from the request body, default to `"BRL"`.

---

## Alternatives Considered

### A. Return 404 if no plan exists; require budget setup first
- **Pros**: Enforces a clean workflow — income should be set via the full budget screen.
- **Cons**: From the Dashboard, a user may not even know they need to create a budget plan. The quick income update is a shortcut precisely for users who haven't gone through the full budget setup. Returning 404 makes the feature useless for new users.
- **Rejected**: Contradicts the UX goal of the feature.

### B. Auto-create plan, but only update `gross_income` — separate endpoint from `POST /budgets` upsert
- **Decision**: This IS the chosen approach. The `UpdateIncomeAsync` method is thin — it reuses `UpsertPlanAsync` internally when no plan exists. The endpoint path `PATCH /budget/income` (rather than `POST /budgets`) communicates the narrower intent to API consumers.

### C. Redirect to existing `POST /budgets` endpoint
- **Pros**: No new endpoint.
- **Cons**: `POST /budgets` requires `month` in the body — unusual ergonomics for a "quick update now" action. The mobile modal only captures a number, not a month selector.
- **Rejected**: Worse mobile UX; the quick income action is intentionally narrow.

---

## Consequences

- **Positive**: The Quick Income modal works for any user, regardless of whether they've set up a budget.
- **Positive**: If a user creates income via the quick modal and then visits the full budget screen, they'll see the plan with correct income and zero allocations — inviting them to add allocations.
- **Negative**: A plan with zero allocations and non-zero income is semantically valid but incomplete. The budget screen's `budget_gap` will equal the full income — this is correct behavior (no allocations yet).
- **Note**: `[RequireCouple]` filter still applies; unauthenticated or couple-less users get 401/403 as usual.

---

## Notes

- Request body: `{ "grossIncome": number, "currency"?: string }`.
- Validator: `grossIncome > 0`; `currency` if present must be 3-letter ISO 4217 code.
- React Query invalidation on success: invalidate `["dashboard"]` and `["budget", "current"]` query keys.
- The modal should close automatically on success; show `toast.success("Renda atualizada.")`.
