# ADR-004: AI Auto-Categorization — Batch vs Individual, Failure Isolation

**Session:** 2026-04-17_v16-bugfixes-bank-import-ux  
**Date:** 2026-04-17  
**Status:** Accepted  
**AC:** AC-612

---

## Context

V1.6 adds optional AI-assisted transaction categorization using the existing Gemini Flash 2.0 integration. The feature must:
- Be failure-safe: if Gemini is unavailable or returns an unparseable response, the import or transaction creation must succeed anyway.
- Avoid prompt injection: transaction descriptions from external sources (bank statement PDFs) are user-controlled text.
- Not significantly increase import latency.
- Reuse the existing `GeminiChatAdapter` / `GeminiOptions` infrastructure.

Two trigger points are specified: (1) OCR import confirmation flow, (2) manual transaction entry.

---

## Decision

### Interface
New `ICategoryClassifier` interface with a single async method:
```csharp
Task<string?> SuggestCategoryAsync(
    string description,
    IReadOnlyList<string> availableCategories,
    CancellationToken ct);
```
Returns `null` on any failure (timeout, Gemini error, parse failure, disabled).

### Implementation
`GeminiCategoryClassifier` in `CoupleSync.Infrastructure.Integrations.Gemini`:
- Calls Gemini with a **structured prompt template**:
  ```
  You are a Brazilian personal finance transaction categorizer.
  Available categories (reply with ONLY one of these, exactly as written, or "other" if none match):
  {category_list}
  
  Transaction description:
  """{description}"""
  
  Reply with only the category name and nothing else.
  ```
- `description` is placed inside `"""` delimiters in a separate prompt section, clearly marked as user data — never interpolated into the instruction segment.
- Response is validated against `availableCategories` list (case-insensitive); if the returned value is not in the list, return `null`.
- All exceptions caught; returns `null`.

### Trigger: OCR import
`OcrProcessingService.ParseAndDeduplicateAsync()` calls `ICategoryClassifier` for each candidate **after** deduplication fingerprinting. The available categories are retrieved from the couple's active budget allocations (category names). If no budget plan exists, a default category list is used (Alimentação, Transporte, Lazer, Saúde, Moradia, Educação, Outros).

### Trigger: Manual transaction entry
`CreateTransactionCommandHandler` (or the service layer equivalent) calls `ICategoryClassifier.SuggestCategoryAsync()` and includes `suggestedCategoryId` in the response. The caller (mobile) decides whether to pre-populate the category field — it is not forced.

### Individual vs batch
**Individual calls per transaction**, not batch, for V1.6. Rationale:
- A typical bank statement has 20–60 transactions. At Gemini Flash free tier (15 RPM, ~1M tokens/day), a 60-transaction import generates 60 requests — within free limits but potentially slow.
- **Sequential with short-circuit**: if Gemini returns two consecutive errors, stop categorizing and return `null` for all remaining candidates (fail-fast to avoid latency accumulation).
- **Timeout**: each call uses a 3-second timeout. If the total batch takes > 15 seconds, abort remaining calls.

---

## Alternatives Considered

### A. Batch all transaction descriptions in a single Gemini call
- **Pros**: One API call; lower latency; lower token overhead per request.
- **Cons**: Response parsing is complex (numbered list, CSV, JSON array — each has edge cases with user-controlled description text). Risk of prompt injection escalates because multiple user inputs are interleaved in one prompt. Harder to attribute results back to specific candidates.
- **Rejected for V1.6**: Reliability burden outweighs latency benefit at pilot scale. Can be revisited in V2 with proper JSON schema–constrained output mode.

### B. Trigger on transaction confirmation only (not during OCR processing)
- **Pros**: Categorization is visible in the review screen before confirmation.
- **Cons**: Requires the review screen to fetch categories async after load, adding a second async step.
- **Decision**: The categorization happens **before** the `OcrCandidate` list is returned to the mobile client (inside `OcrProcessingService`), so the review screen displays pre-populated category chips immediately. This matches AC-612.

### C. No-op classifier always (disable the feature entirely)
- **Pros**: Zero risk.
- **Cons**: AC-612 requires the feature.
- **Retained as fallback**: `NoOpCategoryClassifier` is registered when `AI_CATEGORIZATION_ENABLED=false`.

---

## Consequences

- **Positive**: Categorization failure is completely isolated — it cannot block an import.
- **Positive**: Secure prompt template prevents injection of Gemini system-level instructions through bank statement text.
- **Negative**: 60 sequential Gemini calls per import adds up to 3–15 seconds of background processing time. Users are not waiting on this (processing happens in `OcrBackgroundJob`, not on the request thread), so latency is acceptable.
- **Risk**: If Gemini free-tier RPM limit is hit during a large import, categorization results will be partial (`null` for rate-limited candidates). This is acceptable and consistent with the failure-safe contract.

---

## Notes

- `GeminiCategoryClassifier` reuses `IGeminiAdapter` (existing); no new HTTP client registered.
- `ICategoryClassifier` is registered as `Scoped` to match `OcrProcessingService` lifetime.
- `availableCategories` must be sanitized: trim whitespace, deduplicate, limit to 30 entries before injecting into the prompt.
- Do not log the transaction description at any log level (PII / bank data). Log only `{ transactionIndex, suggestedCategory }` at Debug level.
